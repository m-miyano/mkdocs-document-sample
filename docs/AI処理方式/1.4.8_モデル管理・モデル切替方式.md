# 1.4.8 モデル管理・モデル切替方式

---

## 1. モデル管理方針

LLM モデルは**設定ファイルで管理**し、コード変更なしに切替可能にする。
新モデルの評価・切替はステージング環境で検証してから本番適用する。

---

## 2. モデル設定管理

```yaml
# model_config.yaml（ConfigMap 管理）
models:
  incident_analysis:
    provider: azure_openai
    deployment: gpt-4o-2024-11-20
    fallback_deployment: gpt-4o-mini-2024-07-18
    max_tokens: 2048
    temperature: 0.1          # 再現性重視（低温度）
    timeout_sec: 30

  knowledge_search:
    provider: azure_openai
    deployment: gpt-4o-mini-2024-07-18
    max_tokens: 1024
    temperature: 0.3
    timeout_sec: 15

  embedding:
    provider: azure_openai
    deployment: text-embedding-3-large
    dimensions: 3072
```

## 3. モデルバージョン管理フロー

```mermaid
flowchart TD
    NEW[新モデルリリース情報] --> EVAL[評価環境でベンチマーク実施]
    EVAL --> BENCH{評価結果}
    BENCH -->|改善 ≥ 5%| STAGING[ステージング環境でA/Bテスト]
    BENCH -->|改善 < 5%| HOLD[採用見送り・記録]

    STAGING --> AB{A/Bテスト結果\n2週間}
    AB -->|有意差あり・改善| PROD_CANARY[本番カナリアリリース\n10% トラフィック]
    AB -->|有意差なし or 悪化| ROLLBACK[ステージングに戻す]

    PROD_CANARY --> CANARY_WATCH{モニタリング\n48時間}
    CANARY_WATCH -->|問題なし| FULL_DEPLOY[100% 切替完了]
    CANARY_WATCH -->|異常検知| AUTO_ROLLBACK[自動ロールバック]
```

## 4. フォールバックモデル切替

```mermaid
sequenceDiagram
    participant AGENT as エージェント
    participant ROUTER as Model Router
    participant PRIMARY as GPT-4o（Primary）
    participant FALLBACK as GPT-4o-mini（Fallback）

    AGENT->>ROUTER: LLM リクエスト
    ROUTER->>PRIMARY: リクエスト転送
    
    alt タイムアウト / 429 / 5xx
        PRIMARY-->>ROUTER: エラー
        ROUTER->>ROUTER: フォールバック判定\nエラー種別・連続失敗数
        ROUTER->>FALLBACK: フォールバックリクエスト
        FALLBACK-->>ROUTER: レスポンス
        ROUTER-->>AGENT: レスポンス + fallback_used=true フラグ
    else 正常
        PRIMARY-->>ROUTER: レスポンス
        ROUTER-->>AGENT: レスポンス
    end
```

## 5. モデル評価指標

| 指標 | 測定方法 | 目標値 |
|---|---|---|
| **回答品質スコア** | LLM-as-a-Judge（GPT-4o による採点） | ≥ 4.0 / 5.0 |
| **Faithfulness** | RAGAs フレームワーク | ≥ 0.85 |
| **Answer Relevancy** | RAGAs | ≥ 0.80 |
| **レイテンシ（p95）** | Prometheus | < 5,000ms |
| **コスト（per request）** | トークン数 × 単価 | < ¥2.0 / リクエスト |
